# Screenshot Testing Setup

This directory contains the automated screenshot testing infrastructure for the TestTag for WordPress plugin using Playwright with TypeScript.

## 📁 Directory Structure

```
tests/
├── e2e/
│   └── screenshots.spec.ts          # Main screenshot test suite with all test cases
├── helpers/
│   └── wordpress.ts                 # Reusable WordPress interaction utilities
├── screenshots/                      # Generated screenshots (not committed)
│   ├── 01-settings-page.png
│   ├── 02-attribute-configuration.png
│   └── ... (generated by tests)
└── global-setup.ts                  # One-time setup before all tests run
```

## 🚀 Quick Start

### Windows Users
```bash
cd ci
setup-local.bat
```

### macOS/Linux Users
```bash
bash ci/setup-local.sh
```

Or do it manually:
```bash
# Start services
docker-compose up -d

# Wait a moment for services to be ready
sleep 10

# Install dependencies (includes TypeScript)
npm ci

# Run tests
npm run test:screenshots
```

## 📝 Test Structure

Tests are organized by feature/area:

- **Admin Interface**: Settings pages, configuration panels
- **Frontend**: Element tagging, Audit Mode overlay, user interactions
- **Configuration**: Different attribute types, environment settings

Each test follows this pattern:
1. Log in to WordPress
2. Navigate to specific page
3. Perform interactions if needed
4. Take screenshot(s)
5. Verify elements/content

## 📸 Screenshot Naming Convention

Screenshots are numbered sequentially for easy identification:

- `01-04.png` - Admin/Settings
- `07-09.png` - Frontend/Audit Mode
- `10+.png` - Configuration/other feature tests

## 🔧 Adding New Tests

Add a new test in `e2e/screenshots.spec.ts`:

```typescript
import { test } from '@playwright/test';
import * as wpHelpers from '../helpers/wordpress';

test('Feature name', async ({ page }) => {
  // 1. Set up (login, navigate)
  await wpHelpers.loginToWordPress(page);
  await wpHelpers.goToTestTagSettings(page);
  
  // 2. Interact
  await page.click('button.save');
  await page.waitForLoadState('networkidle');
  
  // 3. Capture
  await page.screenshot({ 
    path: './tests/screenshots/XX-feature-name.png',
    fullPage: true
  });
});
```

## 🎯 Available Helper Functions

See `helpers/wordpress.js` for all available functions:

- `loginToWordPress()` - Authenticate to WordPress admin
- `goToDashboard()`, `goToTestTagSettings()`, `goToBlockEditor()` - Navigation
- `enableAuditMode()`, `disableAuditMode()` - Audit Mode control
- `createTestPage()`, `addBlockEditorContent()` - Page creation
- `waitForTestAttribute()` - Verify element tagging

## 🔄 Updating Screenshots

When UI intentionally changes:

```bash
# Update all baseline screenshots
npm run test:screenshots -- --update-snapshots

# Or specific test
npm run test:screenshots -- tests/e2e/screenshots.spec.js -k "Feature" --update
```

Then commit the updated screenshots to establish the new baseline.

## ❌ Troubleshooting

**Tests timeout:**
- Increase `timeout` in individual test or `global timeout` in config
- Check Docker containers are healthy: `docker-compose ps`

**Plugin not activated:**
- Verify plugin files exist and are readable
- Check setup logs in `global-setup.js`
- Manually verify through WordPress admin

**WordPress not responding:**
- Wait longer: `docker-compose logs wordpress`
- Check MySQL: `docker-compose logs mysql`
- Restart: `docker-compose down -v && docker-compose up -d`

**Screenshots not created:**
- Ensure path is writable: `chmod 755 tests/screenshots`
- Check test didn't fail before screenshot line
- Look for test errors in output
